### 4️⃣ PostgreSQL запросы (5 минут)

#### Показать: **database/crud.py** (строки 30-73)

**Три основных запроса**:

##### Запрос 1: JOIN + GROUP BY (Авторы)
```python
# Файл: database/crud.py, строка 30-41
get_authors_with_article_count()
```
**SQL**:
```sql
SELECT authors.id, authors.name, COUNT(articles.id)
FROM authors
LEFT JOIN articles ON authors.id = articles.author_id
GROUP BY authors.id, authors.name
```

**Назначение**: Показать, сколько статей написал каждый автор
**Результат**: Авторы с количеством своих статей

---

##### Запрос 2: JOIN + GROUP BY (Категории)
```python
# Файл: database/crud.py, строка 44-54
get_categories_with_stats()
```
**SQL**:
```sql
SELECT categories.id, categories.title, COUNT(articles.id)
FROM categories
LEFT JOIN articles ON categories.id = articles.category_id
GROUP BY categories.id, categories.title
```

**Назначение**: Статистика по категориям
**Результат**: Категории с количеством статей в каждой

---

##### ⭐ Запрос 3: ДВОЙНОЙ JOIN + GROUP BY + COUNT(DISTINCT)
```python
# Файл: database/crud.py, строка 57-73
get_issues_with_stats()
```
**SQL**:
```sql
SELECT issues.id, issues.number, issues.publish_date,
       COUNT(articles.id) as article_count,
       COUNT(DISTINCT authors.id) as unique_authors
FROM issues
LEFT JOIN articles ON issues.id = articles.issue_id
LEFT JOIN authors ON articles.author_id = authors.id
GROUP BY issues.id, issues.number, issues.publish_date
ORDER BY issues.publish_date DESC
```

**Особенности**:
- ✓ ДВОЙНОЙ JOIN (Issue → Articles → Authors)
- ✓ GROUP BY
- ✓ COUNT(DISTINCT authors.id) - считаем уникальных авторов
- ✓ ORDER BY DESC
- ✓ SELECT_FROM() - явно указана левая таблица

**Назначение**: Вывести выпуски журнала с информацией о количестве статей и уникальных авторов
**Результат**: Выпуски с расчитанной статистикой

##### Запрос 2: FIND + SORT (Получение комментариев)
```python

                              .sort("created_at", -1)
                              .to_list(100)
```

```javascript

          .sort({created_at: -1})
          .limit(100)
```
**Назначение**: Получить последние комментарии к статье (новые первыми)

---

##### ⭐ Запрос 3: АГРЕГАЦИЯ ($match + $group)
```python
# Файл: utils.py, строка 12-17
cursor = comments_col.aggregate([
    {"$match": {"article_id": article_id}},
    {"$group": {"_id": "$article_id", "avg": {"$avg": "$rating"}}}
])
```
**MongoDB Pipeline**:
```javascript
db.comments.aggregate([
    { $match: { article_id: 1 } },      // Фильтруем
    { $group: {
        _id: "$article_id",
        avg: { $avg: "$rating" }        // Вычисляем среднее
    }}
])
```

**Процесс**:
1. **$match** - фильтруем комментарии статьи
2. **$group** - группируем и считаем среднее значение рейтинга
3. **Результат**: Средний рейтинг статьи (например 4.5)

**Назначение**: Эффективно вычислить средний рейтинг статьи из всех комментариев

### 6️⃣ Redis запросы (4 минуты)

#### Показать: **api/routes.py** + **utils.py**

**Таблица ключей**:

| Ключ | Операция | TTL | Файл | Строка |
|------|----------|-----|------|--------|

| `total:comments` | INCR | Нет | routes.py | 120 |
| `popular:articles` | ZINCRBY | Нет | routes.py | 196 |

---

##### Запрос 1: INCR (Увеличение счетчика)
```python
# Файл: api/routes.py, строка 199
await rd.incr("total:views")
```
**Redis**:
```
INCR total:views
```
**Результат**: `1 → 2 → 3 → ...`
**Назначение**: Считать просмотры всех статей

---

##### Запрос 2: ZINCRBY (Просмотры статьи)
```python
# Файл: api/routes.py, строка 196
await rd.zincrby("popular:articles", 1, str(article_id))
```
**Redis Sorted Set**:
```
ZINCRBY popular:articles 1 "1"
```
**Результат**:
```
"1" → score 251
"5" → score 180
"3" → score 150
```
**Назначение**: Отслеживать просмотры каждой статьи для топа

---

##### ⭐ Запрос 3: SETEX (Кеширование с TTL)
```python
# Файл: utils.py, строка 20
await rd.setex(f"article:{article_id}:avg_rating", 3600, str(avg))
```
**Redis TTL**:
```
SETEX article:1:avg_rating 3600 "4.5"
       └─ Ключ                └──┬── Время жизни (1 час)
                                 └─ Значение (средний рейтинг)
```

**Как это работает**:
- Время 10:00:00 - установлен ключ, истечет в 11:00:00
- Время 10:30:00 - ключ доступен (TTL = 30 мин)
- Время 11:00:00 - ключ истекает (DELETE автоматически)

**Преимущество**: Быстрое получение рейтинга из кеша вместо пересчета из MongoDB

---

##### Запрос 4: DELETE (Инвалидация кеша)
```python
# Файл: api/routes.py, строка 123
await rd.delete(f"article:{article_id}:avg_rating")
```
**Redis**:
```
DEL article:1:avg_rating
```
**Назначение**: Когда добавляется новый комментарий, удалить старый кеш, чтобы рейтинг пересчитался при следующем просмотре

## 🎤 ОТВЕТЫ НА ВОЗМОЖНЫЕ ВОПРОСЫ

### Q1: Почему вы используете три разные БД?
**A**: Каждая БД оптимальна для своего типа данных:
- **PostgreSQL**: Структурированные данные (авторы, статьи) с отношениями
- **MongoDB**: Полу-структурированные данные (комментарии) и статистика
- **Redis**: Кеширование для быстрого доступа к часто запрашиваемым данным

### Q2: Что такое TTL в Redis?
**A**: TTL (Time To Live) - время жизни ключа. После истечения TTL ключ автоматически удаляется. В нашем случае:
- `article:1:avg_rating` существует 3600 секунд (1 час)
- Потом Redis автоматически его удаляет
- При новом просмотре рейтинг пересчитывается

### Q3: Зачем нужен COUNT(DISTINCT)?
**A**: В запросе о выпусках мы считаем уникальных авторов. Если один автор написал несколько статей в выпуске, COUNT(DISTINCT) не будет считать его несколько раз.

### Q4: Почему используется OUTER JOIN?
**A**: OUTER JOIN позволяет найти авторов/категории БЕЗ статей (показать 0). Если бы мы использовали INNER JOIN, пустые авторы не отображались бы.

### Q5: Что такое Aggregation Pipeline в MongoDB?
**A**: Это серия этапов обработки в MongoDB:
1. $match - фильтруем данные
2. $group - группируем и считаем
3. $sort, $limit и т.д.

Похоже на SQL-запрос с GROUP BY.

### Q6: Как работает SELECT_FROM()?
**A**: При множественных JOIN'ах SQLAlchemy не знает, из какой таблицы начать. SELECT_FROM() явно указывает: "Начни с этой таблицы". Это решает ошибку "Can't determine which FROM clause to join from".
