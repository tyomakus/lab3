# Отчет о выполнении лабораторной работы

## Предметная область
Веб-приложение для управления журналом (газетой) с возможностью просмотра статей, добавления комментариев, просмотра рейтингов и статистики.

---

## ЧАСТЬ 1. ВЕБ-ПРИЛОЖЕНИЕ

### Архитектура
- **Фреймворк**: FastAPI (Python)
- **Front-end**: HTML / Jinja2 templates
- **HTTP протокол**: JSON + HTML responses

### Реализованные страницы (6 страниц):
1. **Главная страница** (`/`) - Список всех статей
2. **Страница авторов** (`/authors`) - Список авторов и добавление новых авторов
3. **Добавление статьи** (`/articles/new`) - Форма для создания новой статьи
4. **Деталь статьи** (`/articles/{id}`) - Просмотр статьи, комментарии, рейтинг
5. **Поиск** (`/search`) - Поиск по названию и содержимому с фильтром по рейтингу
6. **Статистика** (`/statistics`) - Общая статистика с JOIN, GROUP BY и агрегацией
7. **Популярное** (`/popular`) - Топ популярных статей из Redis

## ЧАСТЬ 2. БАЗА ДАННЫХ PostgreSQL

### Структура БД (4 таблицы в 3NF):

#### 1. **authors** (Авторы)
```
- id (PRIMARY KEY, Integer)
- name (String, NOT NULL)
- email (String, UNIQUE, NOT NULL)
```

#### 2. **categories** (Категории)
```
- id (PRIMARY KEY, Integer)
- title (String, NOT NULL, UNIQUE)
```

#### 3. **issues** (Выпуски журнала)
```
- id (PRIMARY KEY, Integer)
- number (Integer, NOT NULL)
- publish_date (Date, NOT NULL)
```

#### 4. **articles** (Статьи)
```
- id (PRIMARY KEY, Integer)
- title (String, NOT NULL)
- content (Text, NOT NULL)
- author_id (FOREIGN KEY → authors.id)
- category_id (FOREIGN KEY → categories.id)
- issue_id (FOREIGN KEY → issues.id)
- created_at (TIMESTAMP)
```

### Запросы с JOIN и GROUP BY в `database/crud.py`:

#### 1. **get_authors_with_article_count** (строка 30-41)
```python
SELECT Author.id, Author.name, Author.email, COUNT(Article.id) as article_count
FROM authors
LEFT OUTER JOIN articles ON Author.id = Article.author_id
GROUP BY Author.id, Author.name, Author.email
```
- Показывает каждого автора и количество его статей

#### 2. **get_categories_with_stats** (строка 44-54)
```python
SELECT Category.id, Category.title, COUNT(Article.id) as article_count
FROM categories
LEFT OUTER JOIN articles ON Category.id = Article.category_id
GROUP BY Category.id, Category.title
```
- Показывает статистику по категориям

#### 3. **get_issues_with_stats** (строка 57-73) - МНОГОУРОВНЕВЫЙ JOIN
```python
SELECT Issue.id, Issue.number, Issue.publish_date, COUNT(Article.id), COUNT(DISTINCT Author.id)
FROM issues
LEFT OUTER JOIN articles ON Issue.id = Article.issue_id
LEFT OUTER JOIN authors ON Article.author_id = Author.id
GROUP BY Issue.id, Issue.number, Issue.publish_date
```
- Статистика по выпускам журнала
- Использует SELECT_FROM() для явного указания левой части JOIN
- Содержит COUNT(DISTINCT Author.id) для уникальных авторов

```json

  "total_views": Integer,
  "total_comments": Integer

```

### Агрегация MongoDB (в `utils.py` строка 12-14):
```python
comments_col.aggregate([
    {"$match": {"article_id": article_id}},
    {"$group": {"_id": "$article_id", "avg": {"$avg": "$rating"}}}
])
```
- Вычисляет средний рейтинг статьи
- Использует `$match` для фильтрации
- Использует `$group` с `$avg` для агрегации

## ЧАСТЬ 4. КЕШ Redis

### Ключи Redis:

#### 1. **popular:articles** (Sorted Set с TTL)
- Формат: `ZADD popular:articles <score> <article_id>`
- Использование: Отслеживание количества просмотров каждой статьи
- TTL: Установлен через `EXPIRE` (хранит значения долгое время для популярности)

#### 2. **article{id}:avg_rating** (String с TRUE TTL = 3600 сек)
- Формат: `SETEX article:{id}:avg_rating 3600 <average_rating>`
- Использование: Кеш среднего рейтинга статьи
- **TTL: 3600 секунд (1 час)** - реализовано в `utils.py` строка 20

#### 3. **total:views** и **total:comments** (Integer счетчики)
- Формат: `INCR total:views`, `INCR total:comments`
- Использование: Хранение общего количества просмотров и комментариев

## ИСПРАВЛЕНИЯ ДЛЯ СООТВЕТСТВИЯ ТРЕБОВАНИЯМ

### Проблема 1: Ошибка при выполнении многоуровневого JOIN
**Файл**: `database/crud.py` (строка 64)

**Исходная ошибка**:
```
sqlalchemy.exc.InvalidRequestError: Can't determine which FROM clause to join from
```

**Решение**: Добавлено явное `select_from(Issue)` перед первым JOIN
```python
.select_from(Issue).outerjoin(
    Article, Issue.id == Article.issue_id
).outerjoin(
    Author, Article.author_id == Author.id
)
```
Это исправляет неоднозначность в SQL при наличии нескольких возможных JOIN path'ов

---

### Проблема 2: Статистика (просмотры и комментарии) показывала 0
**Файлы**: `api/routes.py` (строки 200-206, 112-128, 244-260)

**Исходная проблема**:
- Статистика не обновлялась при просмотре статей
- Статистика не обновлялась при добавлении комментариев
- При каждой загрузке страницы статистики показывались только новые значения

**Решение в 2 этапа**:

#### Этап 1: Добавлено обновление счетчиков в Redis
1. При просмотре статьи (`article_detail`, строка 202):
```python
await rd.incr("total:views")
```

2. При добавлении комментария (`add_comment`, строка 117):
```python
await rd.incr("total:comments")
```

#### Этап 2: Объединение старой и новой статистики
В `statistics_page` (строки 244-260) добавлено объединение данных:
```python
# Старые данные из MongoDB
old_views = mongo_stats.get("total_views", 0) if mongo_stats else 0
old_comments = mongo_stats.get("total_comments", 0) if mongo_stats else 0

# Новые данные из Redis
total_views_redis = await rd.get("total:views")
total_comments_redis = await rd.get("total:comments")

# Объединение
mongo_stats = {
    "total_views": old_views + (int(total_views_redis) if total_views_redis else 0),
    "total_comments": old_comments + (int(total_comments_redis) if total_comments_redis else 0)
}
```

**Результат**: Статистика теперь показывает корректные значения, включая данные до и после изменений кода.
