# ЧАСТЬ 1: СТРУКТУРА БАЗЫ ДАННЫХ PostgreSQL

## 1.1 Сущности (4 таблицы)

### Таблица 1: **authors** (Авторы статей)
```sql
CREATE TABLE authors (
    id INTEGER PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL
);
```
**Назначение**: Хранит информацию об авторах статей журнала.

**Поля**:
- `id` - первичный ключ, уникальный идентификатор автора
- `name` - имя автора (не может быть пустым)
- `email` - уникальный адрес электронной почты

---

### Таблица 2: **categories** (Категории статей)
```sql
CREATE TABLE categories (
    id INTEGER PRIMARY KEY,
    title VARCHAR(100) NOT NULL UNIQUE
);
```
**Назначение**: Классификация статей по тематикам.

**Поля**:
- `id` - первичный ключ
- `title` - название категории (уникальное значение)

---

### Таблица 3: **issues** (Выпуски журнала)
```sql
CREATE TABLE issues (
    id INTEGER PRIMARY KEY,
    number INTEGER NOT NULL,
    publish_date DATE NOT NULL
);
```
**Назначение**: Информация о выпусках журнала (номер выпуска и дата публикации).

**Поля**:
- `id` - первичный ключ
- `number` - номер выпуска журнала
- `publish_date` - дата выхода выпуска

---

### Таблица 4: **articles** (Статьи журнала)
```sql
CREATE TABLE articles (
    id INTEGER PRIMARY KEY,
    title VARCHAR(200) NOT NULL,
    content TEXT NOT NULL,
    author_id INTEGER FOREIGN KEY REFERENCES authors(id),
    category_id INTEGER FOREIGN KEY REFERENCES categories(id),
    issue_id INTEGER FOREIGN KEY REFERENCES issues(id),
    created_at TIMESTAMP
);
```
**Назначение**: Основная таблица с содержимым статей журнала.

**Поля**:
- `id` - первичный ключ
- `title` - заголовок статьи
- `content` - текст статьи
- `author_id` - ссылка на таблицу authors
- `category_id` - ссылка на таблицу categories
- `issue_id` - ссылка на таблицу issues
- `created_at` - время создания статьи

---

## 1.2 Связи между сущностями

```
authors (1) ──→(*) articles
  ├─ One author can write MANY articles
  └─ FK: articles.author_id → authors.id

categories (1) ──→ (*) articles
  ├─ One category contains MANY articles
  └─ FK: articles.category_id → categories.id

issues (1) ──→ (*) articles
  ├─ One issue contains MANY articles
  └─ FK: articles.issue_id → issues.id
```

**Отношения**: One-to-Many (1:M)
- Один **автор** может написать несколько **статей**
- Одна **категория** может содержать несколько **статей**
- Один **выпуск** может содержать несколько **статей**

**Целостность данных**:
- Foreign Keys обеспечивают ссылочную целостность
- Все таблицы находятся в 3NF (нет дублирования данных)

```python
async def get_authors_with_article_count(db: AsyncSession):
    stmt = select(
        Author.id,
        Author.name,
        Author.email,
        func.count(Article.id).label('article_count')
    ).select_from(Author).outerjoin(
        Article, Author.id == Article.author_id
    ).group_by(Author.id, Author.name, Author.email)

    result = await db.execute(stmt)
    return result.all()
```

**SQL эквивалент**:
```sql
SELECT
    authors.id,
    authors.name,
    authors.email,
    COUNT(articles.id) as article_count
FROM authors
LEFT OUTER JOIN articles ON authors.id = articles.author_id
GROUP BY authors.id, authors.name, authors.email;
```

**Как это работает**:
1. `SELECT_FROM(Author)` - явно указываем левую таблицу (Authors)
2. `OUTERJOIN(Article)` - присоединяем статьи (LEFT JOIN)
3. `GROUP BY` - группируем по авторам
4. `COUNT(Article.id)` - считаем статьи каждого автора

### ⭐ Запрос 3: ДВОЙНОЙ JOIN + GROUP BY + COUNT(DISTINCT)
**Файл**: `database/crud.py`, строка 57-73
**Описание**: Статистика по выпускам журнала с уникальными авторами

```python
async def get_issues_with_stats(db: AsyncSession):
    stmt = select(
        Issue.id,
        Issue.number,
        Issue.publish_date,
        func.count(Article.id).label('article_count'),
        func.count(func.distinct(Author.id)).label('unique_authors')
    ).select_from(Issue).outerjoin(
        Article, Issue.id == Article.issue_id
    ).outerjoin(
        Author, Article.author_id == Author.id
    ).group_by(
        Issue.id, Issue.number, Issue.publish_date
    ).order_by(desc(Issue.publish_date))

    result = await db.execute(stmt)
    return result.all()
```

**SQL эквивалент**:
```sql
SELECT
    issues.id,
    issues.number,
    issues.publish_date,
    COUNT(articles.id) as article_count,
    COUNT(DISTINCT authors.id) as unique_authors
FROM issues
LEFT OUTER JOIN articles ON issues.id = articles.issue_id
LEFT OUTER JOIN authors ON articles.author_id = authors.id
GROUP BY issues.id, issues.number, issues.publish_date
ORDER BY issues.publish_date DESC;
```

**Особенности**:
- **SELECT_FROM(Issue)** - явно указываем начальную таблицу (важно!)
- **ДВОЙНОЙ JOIN**: Issue → Articles → Authors
- **COUNT(DISTINCT Author.id)** - считаем УНИКАЛЬНЫЕ авторов (без дубликатов)
- **ORDER BY DESC** - сортируем по дате выхода (новые выпуски первыми)

**Результат**:
```
id | number | publish_date | article_count | unique_authors
---|--------|--------------|---------------|--------------
3  | 12     | 2024-12-15   | 8             | 4
2  | 11     | 2024-11-15   | 6             | 3
1  | 10     | 2024-10-15   | 5             | 3
```

**Где используется**: На странице `/statistics` для вывода статистики выпусков

### ⭐ Запрос 3: АГРЕГАЦИЯ (Средний рейтинг статьи)
**Файл**: `utils.py`, строка 12-17
**Описание**: Вычисление среднего рейтинга статьи из всех комментариев

```python
# Агрегация для вычисления среднего рейтинга
cursor = comments_col.aggregate([
    {"$match": {"article_id": article_id}},
    {"$group": {"_id": "$article_id", "avg": {"$avg": "$rating"}}}
])
stats = await cursor.to_list(1)
avg = stats[0]["avg"] if stats else 0
```

**MongoDB Aggregation Pipeline**:
```javascript
db.comments.aggregate([
    {
        $match: { article_id: 1 }  // Фаза 1: фильтруем комментарии статьи 1
    },
    {
        $group: {                    // Фаза 2: группируем и вычисляем
            _id: "$article_id",
            avg: { $avg: "$rating" }
        }
    }
]);
```

**Как это работает**:

| Этап | Описание | Данные |
|------|---------|--------|
| **$match** | Фильтруем комментарии статьи с id=1 | rating: [5, 4, 5, 3, 4] |
| **$group** | Вычисляем среднее значение | avg = (5+4+5+3+4)/5 = 4.2 |

**Результат**:

```json
}
```

## 3.3 ЗАПРОСЫ К REDIS

### ⭐ Запрос 1: INCR (Увеличение счетчика просмотров)
**Файл**: `api/routes.py`, строка 199
**Описание**: Увеличение общего счетчика просмотров на 1

```python
# GET /articles/{article_id}
await rd.incr("total:views")
```

**Redis операция**:
```
INCR total:views
```

**Как это работает**:
1. Если ключ не существует, Redis создает его со значением 1
2. Если ключ существует, увеличивает значение на 1
3. Возвращает новое значение

**Пример выполнения**:
```
$ INCR total:views  → 1
$ INCR total:views  → 2
$ INCR total:views  → 3
$ GET total:views   → "3"
```

**Где используется**: При каждом просмотре статьи увеличиваем счетчик

### ⭐ Запрос 3: SETEX (Кеширование с TTL)
**Файл**: `utils.py`, строка 20
**Описание**: Кеширование среднего рейтинга статьи с временем жизни 3600 секунд (1 час)

```python
# Кешируем в Redis с TTL (3600 секунд = 1 час)
await rd.setex(cache_key, 3600, str(avg) if avg > 0 else "null")
```

**Redis операция**:
```
SETEX article:1:avg_rating 3600 "4.2"
```

**Как это работает**:
- `SETEX` = SET + EXPIRE (установить значение и время жизни)
- `article:1:avg_rating` - ключ (кэш для статьи 1)
- `3600` - время жизни в секундах (1 час)
- `"4.2"` - среднее значение рейтинга

**Пример жизненного цикла**:
```
Время 10:00:00 - SETEX article:1:avg_rating 3600 "4.2"
               └─ Ключ создан, истечет в 11:00:00

Время 10:30:00 - GET article:1:avg_rating
               └─ Возвращает "4.2", TTL = 1800 сек оставалось

Время 11:00:00 - GET article:1:avg_rating
               └─ Возвращает nil (ключ истек, нужно пересчитать)
```

**Где используется**: На странице статьи для быстрого получения среднего рейтинга без обращения к MongoDB


**Результат**:
```
id | number | publish_date | article_count | unique_authors
---|--------|--------------|---------------|--------------
3  | 12     | 2024-12-15   | 8             | 4
2  | 11     | 2024-11-15   | 6             | 3
1  | 10     | 2024-10-15   | 5             | 3
```

---

### MongoDB решение (АГРЕГАЦИЯ)
```javascript
db.articles.aggregate([
    {
        $group: {
            _id: "$issue_id",
            issue_number: { $first: "$issue_number" },
            publish_date: { $first: "$publish_date" },
            article_count: { $sum: 1 },
            unique_authors: { $addToSet: "$author_id" }
        }
    },
    {
        $addFields: {
            unique_authors_count: { $size: "$unique_authors" }
        }
    },
    {
        $sort: { publish_date: -1 }
    }
]);
```

**Результат**:
```json
{
  "_id": 3,
  "issue_number": 12,
  "publish_date": "2024-12-15",
  "article_count": 8,
  "unique_authors": [1, 2, 3, 4],
  "unique_authors_count": 4
}
```

## Сравнение подходов

| Аспект | SQL + GROUP BY | MongoDB + Aggregation |
|--------|---|---|
| **Структура** | Табличная (строки/столбцы) | Документная (JSON) |
| **Производительность** | Оптимизирована для JOIN | Требует конвертации |
| **Код** | Декларативный | Процедурный (pipeline) |
| **Масштабируемость** | Хорошо для 1NF, 2NF, 3NF | Хорошо для денормализованных данных |
| **Использование** | Структурированные данные | Полу-структурированные данные |
