# ЧАСТЬ 1: БАЗА ДАННЫХ

## 1.1 Инфологическое проектирование БД

### Таблица 1: Сущности и связи

| Сущность 1 | Сущность 2 | Тип связи | Описание связи |
|---|---|---|---|
| **Authors** | **Articles** | 1:M (One-to-Many) | Один автор может написать несколько статей. Статья должна быть написана одним автором |
| **Categories** | **Articles** | 1:M (One-to-Many) | Одна категория может содержать несколько статей. Каждая статья принадлежит одной категории |
| **Issues** | **Articles** | 1:M (One-to-Many) | Один выпуск журнала может содержать несколько статей. Каждая статья относится к одному выпуску |
| **Articles** | **Comments** (MongoDB) | 1:M (Логическая связь) | Одна статья может иметь несколько комментариев. Связь хранится через поле article_id в MongoDB |

## 1.5 Описание ключей Redis

| Название ключа | Тип данных | Время жизни (TTL) | Описание |
|---|---|---|---|
| `total:views` | String (Integer) | Нет ограничений | Счетчик общего количества просмотров всех статей |
| `total:comments` | String (Integer) | Нет ограничений | Счетчик общего количества комментариев |
| `popular:articles` | Sorted Set | Нет ограничений | Отсортированный набор статей с их количеством просмотров (member→score) |
| `article:{id}:avg_rating` | String (Float) | **3600 секунд (1 час)** | Закешированный средний рейтинг статьи. TTL = 3600 сек - после истечения ключ удаляется и рейтинг пересчитывается |

**Примеры значений**:
```
total:views = "1251"
total:comments = "341"

popular:articles = {
  "1": 251,    // Статья 1 имеет 251 просмотр
  "5": 180,    // Статья 5 имеет 180 просмотров
  "3": 150,    // Статья 3 имеет 150 просмотров
  "2": 120,    // Статья 2 имеет 120 просмотров
  "7": 95      // Статья 7 имеет 95 просмотров
}

article:1:avg_rating = "4.5"  (expires in 3600s)
article:2:avg_rating = "4.2"  (expires in 3600s)
```

#### 5. GET `/search?q=текст&min_rating=3`
**Описание**: Поиск статей по названию/содержимому с фильтром по рейтингу

**Входные параметры**:
- `q` (query parameter) - поисковая фраза (обязательно)
- `min_rating` (query parameter) - минимальный рейтинг (опционально, по умолчанию 0)

**Процесс**:
1. Поиск в PostgreSQL с использованием ILIKE (case-insensitive)
2. Для каждой найденной статьи вычисляется средний рейтинг (из Redis кеша или MongoDB)
3. Фильтруются результаты по min_rating

**Выходные параметры**:
- HTML страница (search.html)
- Результаты:
  ```
  {
    results: [
      {
        article: {...},
        avg_rating: 4.5
      },
      ...
    ],
    q: "текст"
  }
  ```

### POST запросы

#### 1. POST `/authors/new`
**Описание**: Создать нового автора

**Входные параметры** (Form data):
- `name` (string, обязательно) - имя автора
- `email` (string, обязательно) - email автора (должен быть уникальным)

**Пример запроса**:
```html
<form method="POST" action="/authors/new">
  <input name="name" value="Александр Новиков" />
  <input type="email" name="email" value="alex@mail.ru" />
  <button type="submit">Добавить автора</button>
</form>
```

**Процесс**:
1. Валидация входных данных
2. Проверка уникальности email
3. INSERT в таблицу authors
4. Redirect на `/authors`

**Выходные параметры**:
- HTTP 303 Redirect на `/authors`
- Новый автор добавлен в БД

---

#### 2. POST `/articles/new`
**Описание**: Создать новую статью

**Входные параметры** (Form data):
- `title` (string, обязательно) - заголовок (макс 200 символов)
- `content` (string, обязательно) - содержание статьи

**Процесс**:
1. Валидация входных данных

4. Установка автоматического created_at (текущее время)
5. Redirect на `/`

**Выходные параметры**:
- HTTP 303 Redirect на `/`

#### 3. POST `/articles/{article_id}/comments`
